? 문제 상황 파악 — 언제 이 가이드가 필요한가

다음 증상이 하나라도 해당되면 이 가이드를 따른다.

  • Obsidian 내장 터미널에서 claude / opencode / nvm / pip 등이 "command not found"
  • 일반 터미널(gnome-terminal, kitty 등)에서는 동일 명령이 정상 동작
  • LLM 도구(opencode + GLM-5 등 중국산 모델)가 중국어로 thinking/응답
  • .bashrc alias, 함수가 Obsidian 터미널에서만 미적용
근본 원인: Obsidian은 Electron 앱으로, 터미널 플러그인이 생성하는 쉘이 login shell이 아닌 /bin/sh로 실행된다. sh는 ~/.bashrc·~/.profile을 로드하지 않아 사용자가 설정한 PATH가 전혀 반영되지 않는다.

1 터미널 플러그인 프로파일 수정 — 핵심

파일: <vault-root>/.obsidian/plugins/terminal/data.json
profiles.linuxIntegratedDefault 객체에서 두 필드를 변경한다.

// Before (기본값) "linuxIntegratedDefault": { "args": [], "executable": "/bin/sh", ... } // After (수정) "linuxIntegratedDefault": { "args": ["--login", "-i"], "executable": "/bin/bash", ... }

각 플래그의 역할:

--login → /etc/profile → ~/.profile → ~/.bashrc 로드 → PATH 완전 구성
-i → interactive 모드 → alias · 함수 활성화
macOS: Darwin 프로파일은 /bin/zsh에 ["--login"]이 이미 설정되어 있어 별도 수정 불필요. Windows: cmd.exe 기반이므로 이 가이드와 무관.

수정 후 Obsidian에서 플러그인을 재활성화해야 적용된다:

  • Settings → Community plugins → Terminal 비활성화 → 재활성화
  • 또는 Obsidian 완전 재시작

2 로케일 환경변수 보장 — AI 도구 언어 고정

파일: ~/.bashrc
Electron 앱은 OS 세션의 locale 환경변수를 그대로 상속하지 않는 경우가 있다. GLM-5처럼 중국산 LLM은 LANG/LC_ALL이 없으면 중국어로 defaulting한다.

# ~/.bashrc 상단, interactive check(case $- in) 이전에 추가 # Locale — Electron 터미널에서 AI 도구가 중국어 defaulting하는 것 방지 export LANG=en_US.UTF-8 export LC_ALL=en_US.UTF-8
왜 interactive check 앞에 넣어야 하는가:
bash의 기본 ~/.bashrc는 case $- in *i*) 체크 이후 return한다. check 뒤에 export를 두면 non-interactive 쉘(스크립트, opencode 내부 호출 등)에서 적용되지 않는다. check 앞에 두면 모든 bash 호출에서 무조건 실행된다.
이미 시스템 locale이 /etc/default/locale에 올바르게 설정돼 있더라도, Electron 프로세스가 /etc/profile.d/를 거치지 않고 실행될 수 있으므로 .bashrc에 명시적으로 export하는 것이 안전하다.

3 단축 명령 스크립트 생성 (선택)

긴 명령을 짧게 쓰고 싶을 때. alias 대신 실행 스크립트를 ~/.local/bin/에 두면 어떤 shell에서도 동작한다.

예시: cc → claude --dangerously-skip-permissions

# 생성 cat > ~/.local/bin/cc <<'EOF' #!/bin/bash exec claude --dangerously-skip-permissions "$@" EOF chmod +x ~/.local/bin/cc # 확인 which cc # → /home/user/.local/bin/cc
alias vs 스크립트: alias는 ~/.bashrc가 로드된 interactive bash에서만 동작. 스크립트는 PATH에 ~/.local/bin이 있으면 sh, bash, 어떤 subshell에서도 동작. exec를 쓰면 wrapper 프로세스를 남기지 않고 바로 대체된다.

Obsidian이 snap 패키지로 설치되어 있어서, 실행 시 XDG 표준 경로를 snap 전용 격리 경로로 override한다. 터미널 플러그인이 띄운 쉘은 이 환경을 그대로 상속한다.

# Obsidian(snap) 프로세스 환경변수 XDG_DATA_HOME = /home/ducky/snap/obsidian/62/.local/share ← snap 격리 XDG_CONFIG_HOME = /home/ducky/snap/obsidian/62/.config ← snap 격리 # 정상 값 XDG_DATA_HOME = /home/ducky/.local/share XDG_CONFIG_HOME = /home/ducky/.config

opencode는 API 인증 키를 $XDG_DATA_HOME/opencode/auth.json에서 읽는다.

# snap 경로의 auth.json (Obsidian 터미널이 읽는 파일) { "zhipuai": { "type": "api", "key": "..." } } ↑ 구버전 provider 키 # 실제 사용해야 할 auth.json (~/.local/share/opencode/auth.json) { "zai-coding-plan": { "type": "api", "key": "..." } } ↑ opencode.json의 model: "zai-coding-plan/glm-5" 와 일치
결과
잘못된 provider 키(zhipuai)로 api.z.ai에 요청 → HTTP 429 (인증 실패 / 과부하로 위장) → GLM-5가 thinking 스트리밍 중 끊김 → 중국어 thinking만 보이고 최종 답변 없음
왜 일반 터미널에서는 되는가
일반 터미널은 snap 환경 밖에서 실행 → XDG 경로가 표준값 유지 → ~/.local/share/opencode/auth.json에서 zai-coding-plan 키를 올바르게 읽음

수정 — .bashrc에서 XDG 경로 강제 복원
핵심 수정 1줄 추가
~/.bashrc ▼

snap이 XDG 경로를 process 레벨에서 오염시키지만, bash가 시작할 때 .bashrc에서 재정의하면 덮어쓸 수 있다. 반드시 interactive check(case $- in) 앞에 위치해야 한다.

~/.bashrc
export LANG=en_US.UTF-8
export LC_ALL=en_US.UTF-8
+ # Override snap's XDG path hijacking so tools like opencode use correct auth/config
+ export XDG_DATA_HOME="$HOME/.local/share"
+ export XDG_CONFIG_HOME="$HOME/.config"
# 직접 편집하거나 아래 명령으로 추가 cat >> ~/.bashrc <<'EOF' # Override snap's XDG path hijacking so tools like opencode use correct auth/config export XDG_DATA_HOME="$HOME/.local/share" export XDG_CONFIG_HOME="$HOME/.config" EOF
왜 이 방법이 작동하는가:
snap은 XDG 경로를 Obsidian 프로세스 시작 시점에 설정한다. bash가 child process로 실행될 때 이 값을 상속하지만, .bashrc가 실행되면서 export로 덮어쓰면 이후 bash에서 실행되는 모든 프로그램(opencode 포함)은 올바른 경로를 본다.
주의: XDG_DATA_HOME을 변경하면 snap으로 설치된 다른 앱이 데이터를 ~/.local/share에 쓸 수 있다. 대부분의 경우 문제없지만, snap 앱의 데이터 격리를 의도적으로 사용하고 있다면 영향을 확인할 것.
